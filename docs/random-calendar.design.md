# Random Calendar Design

## 1. Document purpose

本文档描述 random question 页面当前已经沉淀下来的产品设计结果。

它不是开发备忘，也不是接口字段罗列，而是对以下问题给出统一设计结论：

- 这个页面到底帮助用户完成什么任务
- 单日查看与范围规划之间如何分工
- 日历视图应该表达哪些状态，不应该表达哪些状态
- 规划数据、已保存数据、临时预览数据之间如何切换
- 当缓存、并发、版本变化等真实业务问题出现时，页面应如何表现

本文档只描述当前已确认的设计，不保留早期试验方案。

---

## 2. Product goal

random question 页面的核心目标，不是“选日期”，而是帮助用户在一个连续时间轴上完成题组安排。

这个安排流程分成三类任务：

1. 查看某一天已经保存的题组
2. 为某一天临时生成一组候选题组并决定是否保存
3. 为一段连续日期预先生成多天题组，并逐天审核、逐天保存或批量保存

因此这个页面必须同时支持：

- 精确的单日操作
- 高效率的范围规划
- 稳定的临时结果审查
- 可控的批量提交

这也是为什么当前设计不再把范围选择直接塞进月历本体，而是拆成独立的规划交互。

---

## 3. Core design principles

## 3.1 Day-first, range-second

页面的最终落点永远是“某一天的题组结果”。

即使用户先进行了范围选择，最后他看到的、审查的、保存的，仍然是具体日期上的具体题组。

范围选择只是为了提升规划效率，不是最终结果形态。

## 3.2 Calendar shows state, not workflow history

月历本身只负责表达当前日期状态，不负责保留交互过程。

也就是说：

- 月历可以显示某天已有保存结果
- 月历可以显示某天已有临时规划结果
- 月历可以显示当前选中日期

但月历不应长期保留“刚才拖过的范围”这类过程态。

范围只是生成规划的输入，规划结果才是页面需要持续表达的状态。

## 3.3 Planning must remain stable during review

用户一旦拿到一批临时规划数据，这批数据在他审查期间必须保持稳定。

否则会出现非常差的体验：

- 用户刚看完第 1 天
- 再看第 2 天时，题组内容已经悄悄变了
- 或者单日保存了一条后，其他天的临时规划全部失效

因此当前设计明确要求：

- 规划结果一旦生成，就作为一批稳定的临时数据存在
- 用户可以逐天处理，也可以批量处理
- 在用户主动刷新之前，这批规划结果不应被重新计算替换

## 3.4 Refresh is an explicit operation

题组分析和规划都依赖一份全局计算结果。

这份结果可能因为题库变化或其他操作而过期，但系统不应静默替换用户正在审查的数据。

因此刷新必须是一个显式动作：

- 用户主动刷新，接受新的分析结果
- 页面再基于新结果重新生成后续规划

这能避免“用户正在处理一批计划时，系统后台悄悄换了一批新数据”的不确定性。

## 4. Page structure

页面由两个主要区域组成：

1. 左侧日历视图
2. 右侧信息与操作视图

两者不是主次关系，而是职责分工：

- 左侧负责时间定位和状态概览
- 右侧负责当前日期的内容、分析和补充信息

---

## 5. Calendar view

## 5.1 Calendar responsibilities

日历视图承担四个职责：

1. 选择当前查看日期
2. 展示哪些日期已有已保存题组
3. 展示哪些日期已有临时规划题组
4. 提供进入范围规划或批量处理的入口

它不承担以下职责：

- 不直接承载范围拖选的全过程
- 不长期显示范围背景
- 不直接承担批量保存确认

## 5.2 Date status semantics

当前月历中的每个日期格只表达有限状态，保持语义清晰。

### Selected

表示当前右侧详情区域正在围绕这一天展开。

这个状态是页面的当前焦点。

### Saved

表示这一天已经存在正式保存的题组结果。

点击后应进入该日期的已保存详情查看。

### Planned

表示这一天已经有临时规划结果，但尚未正式保存。

点击后应直接查看该日期对应的临时规划内容，而不是重新生成。

### Empty

表示这一天当前没有已保存结果，也没有临时规划结果。

点击只改变当前焦点日期，不触发无意义详情请求。

## 5.3 Calendar navigation model

月历导航采用“日期迁移”而不是“纯视图翻页”。

这意味着：

- 当用户当前选中某一天时
- 点击上月、下月、去年、明年
- 选中的日期也应随之迁移

例如：

- 3 月 8 日点击下月后，应变成 4 月 8 日
- 3 月 8 日点击去年后，应变成上一年的 3 月 8 日

如果目标月份没有该日，则自动收敛到该月最后一天。

这样设计的原因是：

- 用户关心的是“这一天在别的月份/年份对应的位置”
- 而不是单纯把视角翻到另一个月后再重新找日期

这种模型对连续排期和多月审查更自然。

## 5.4 Current day shortcut

月历提供一个“快速回到今天”的能力。

它的语义不是仅仅回到今天所在月份，而是：

- 直接将今天设置为当前选中日期
- 同时让月历回到今天所在月份

这是一种时间定位捷径。

## 5.5 Range and batch action entry

日历头部只保留一个与规划相关的主入口，但它会根据当前页面状态切换职责。

### When no planned data exists

入口用于发起新的范围规划。

用户通过它进入范围选择弹窗。

### When planned data exists

入口用于处理当前这批临时规划结果。

用户通过它进入批量处理弹窗，而不是继续发起新规划。

这样设计的目的是：

- 页面在同一时刻只维护一批临时规划数据
- 避免用户还未处理完当前规划，就再次生成新的规划批次
- 保证当前正在审查的计划集合边界清晰

## 6. Range planning dialog

## 6.1 Why a separate dialog

范围规划被设计成独立弹窗，而不是直接嵌进月历，有三个原因：

1. 范围选择本质上是过渡操作，不是页面主状态
2. 移动端需要完整遮罩和独立操作空间
3. 月历本体更适合表达结果状态，不适合承载复杂拖动交互

因此，范围规划应被视为一个“生成规划输入”的独立工作区。

## 6.2 Reference date

范围规划始终围绕一个参考日期展开。

当前设计中，参考日期的来源规则非常明确：

- 默认参考日期来自日历当前选中的日期
- 如果用户还没有明确选中过其他日期，则可回退到今天

这个参考日期是范围弹窗内部所有快捷操作的基础，包括：

- 当前日
- 最近若干天
- 本月
- 月份切换
- 年份切换

换言之，弹窗不是以“当前显示月份”为中心，而是以“当前参考日期”为中心。

## 6.3 Range interaction model

范围选择器使用轨道式交互，而不是传统双月面板。

它支持三类操作：

1. 拖动左侧控制点，修改开始日期
2. 拖动右侧控制点，修改结束日期
3. 拖动中间选中区域，整体平移当前窗口

这样设计的原因是：

- 自定义起止日期时，左右边界调整更直接
- 已经形成固定长度窗口后，整体平移更高效
- 对移动端和桌面端都更容易形成一致交互

## 6.4 Double-click reset

轨道支持通过双击重新选择参考点。

它的语义不是直接生成结果，而是：

- 将参考日期重置到用户双击的位置
- 并基于该点重新建立单日窗口

这是为了让用户在长时间轴上快速跳到新的时间位置。

## 6.5 Quick ranges

快捷范围不是独立的时间模板，而是围绕当前参考日期展开的窗口捷径。

例如：

- 7D 表示从当前参考日期开始的 7 天窗口
- 10D、15D、30D 同理

本月是一个特殊快捷项：

- 它取当前参考日期所在自然月的完整起止边界

因此，快捷范围的核心不是固定时间段，而是“基于当前参考点的快速展开”。

## 6.6 Range result presentation

弹窗中会实时显示当前选择结果：

- 起止日期文本
- 当前窗口覆盖天数

在拖动过程中，这些结果应实时更新。

原因很简单：

- 用户在拖动时关心的不只是位置
- 还关心自己到底选中了多少天、起止落在哪里

这属于交互反馈，不应延迟到松手之后才更新。

## 6.7 Apply semantics

点击确认后，范围弹窗的职责结束。

此时页面应发生以下变化：

1. 根据范围请求规划结果
2. 将返回的规划结果映射到具体日期上
3. 在月历中为这些日期标记“已规划”
4. 自动切换到第一条规划日期
5. 右侧详情区域直接展示该日的临时规划内容

重点在于：

- apply 后，页面保留的是规划结果
- 而不是范围选择过程

也就是说，弹窗关闭后，范围态本身不应继续占据主界面语义。

## 7. Planning workflow

## 7.1 Single-day preview

对于单个日期，用户可以请求生成一组临时候选题组。

这组结果属于预览或临时规划态，尚未正式保存。

用户可以：

- 保存这一天
- 取消这一天的临时结果
- 若当天已有正式数据，则以替换方式保存

## 7.2 Multi-day planning

对于一段连续日期，系统会一次性生成多天的临时规划结果。

这些结果形成一批待处理计划，用户后续可以：

- 逐天查看
- 逐天保存
- 逐天取消某个日期
- 一次性批量保存全部可提交日期
- 一次性清除整批临时规划

## 7.3 One active planning batch

当前设计只允许页面同时存在一批有效的临时规划数据。

这样做不是限制能力，而是降低歧义。

如果允许多批规划交错存在，会立刻引出复杂问题：

- 哪一批是当前有效规划
- 批量保存作用于哪一批
- 再次生成时是否要保留旧规划
- 日历上的临时标记属于哪次操作

因此，页面当前坚持“单批规划”的模型。

## 7.4 Generate guard

如果页面上已经存在尚未处理完的临时规划数据，用户不能直接再次发起新的生成。

系统会要求用户先回到当前规划结果进行处理。

这样做的目的是：

- 避免重复取用同一批底层可用题组
- 避免把当前审查流程打断
- 保证用户眼前这批计划是连续、完整、可追踪的

## 8. Frontend component structure

当前前端实现把“日历视图能力”“滑动范围选择能力”“random question 业务编排”拆成三层。

这个拆分的目标不是简单缩短页面文件，而是让两类通用能力可以被其他业务复用：

- 单独使用滑动窗口日期组件
- 单独使用带状态的日历视图组件
- 在具体业务中组合两者，形成“范围准备数据 + 单日审查操作 + 批量处理”的完整流程

## 8.1 CalendarStatusView

`CalendarStatusView` 是通用的带状态日历视图组件。

如果不叠加任何其他业务逻辑，单独使用它，本质上等价于：

- 一个可切换选中日期的月历
- 一个能显示日期状态的月历
- 一个带单个主操作按钮的月历

也就是说，它只是“日历视图本身”，不是完整业务面板。

它沉淀的能力包括：

1. 显示固定 42 格的月历视图
2. 支持选中日期
3. 支持日期迁移式导航
4. 支持顶部六个按钮布局
5. 支持单日点击后切换当前选中日期
6. 支持每一天的数据状态展示
7. 支持一个由使用方配置的主操作按钮

它不包含 random question 业务逻辑。

也就是说，它不应该知道：

- 哪些日期是题组已保存
- 哪些日期是题组已规划
- 点击范围按钮后是否要请求 `/api/random-questions/plan-range`
- 是否存在 `snapshotVersion`
- 是否要批量保存

这些都属于业务容器职责。

`CalendarStatusView` 只接收已经整理好的日历输入：

- `selectedDate`
- `dayStates`
- `action`
- `onSelectedDateChange`

其中，`action` 表示顶部中间的业务主入口。

在 random question 页面中，这个入口会根据当前状态切换：

- 没有规划数据时，显示滑动日期选择入口
- 已有规划数据时，显示批量处理入口

但这个切换本身不属于通用日历组件。

## 8.2 Calendar day state

日历中的日期状态通过 `dayStates` 传入。

它的设计重点是：日历只负责表达状态，不负责解释业务。

每个日期状态至少包含一个业务 key：

```ts
type CalendarDayState<TStateKey extends string = string> = {
  key: TStateKey;
  title?: string;
  tone?: 'saved' | 'planned' | 'warning' | 'danger' | 'neutral';
};
```

`key` 由业务方定义。

例如 random question 页面中使用：

- `saved`
- `planned`

其他业务可以使用：

- `published`
- `draft`
- `scheduled`
- `blocked`

`tone` 只用于视觉表达。它不是业务枚举，也不应该反向驱动业务逻辑。

单日点击时，通用日历组件只负责切换选中日期：

```tsx
<CalendarStatusView
  selectedDate={selectedDate}
  dayStates={dayStates}
  onSelectedDateChange={setSelectedDate}
  action={calendarAction}
/>
```

这保证了一个核心交互原则：

- 日历上的每一天都是可操作入口
- 不同状态只影响视觉表达与页面后续展示数据
- 范围选择或批量处理不会取代单日操作

如果业务需要“点击某天后联动请求数据、切换详情区、打开特定弹窗、驱动批处理流程”，这些都不属于 `CalendarStatusView` 自身职责，而应由业务方在外层封装。

## 8.3 RandomDateRangeDialog

`RandomDateRangeDialog` 是独立的滑动窗口日期组件。

如果不叠加任何其他业务逻辑，单独使用它，本质上等价于：

- 一个日期范围选择弹窗
- 一个输出 `startDate/endDate` 的滑动窗口选择器

也就是说，它只是“范围输入器”，不是完整规划流程。

它只负责范围选择交互，输出：

- `startDate`
- `endDate`

它不关心：

- 范围选择后请求哪个接口
- 范围选择后生成什么数据
- 是否要在日历上标记 planned
- 是否允许批量操作

因此它可以被筛选器、报表、导出、排期等业务单独使用。

单独使用时，调用方只需要维护 open 状态和 range 状态：

```tsx
<RandomDateRangeDialog
  open={rangeOpen}
  value={range}
  anchorDate={selectedDate}
  defaultRangeDays={7}
  onOpenChange={setRangeOpen}
  onApply={(nextRange) => {
    setRange(nextRange);
    runQuery(nextRange);
  }}
/>
```

## 8.4 RandomQuestionCalendarPanel

`RandomQuestionCalendarPanel` 是 random question 页面专用的业务容器。

它组合：

- `CalendarStatusView`
- `RandomDateRangeDialog`

并负责把 random question 的业务状态映射到通用日历组件。

它承担的职责包括：

1. 把已保存日期映射为 `saved` day state
2. 把临时规划日期映射为 `planned` day state
3. 根据是否存在 planned days 切换顶部主操作按钮
4. 打开滑动日期范围弹窗
5. 调用范围规划接口
6. 把接口返回的 planned dates 映射为页面的 `PlannedDay`
7. 在范围规划成功后切换到第一条规划日期
8. 把 preview state、snapshot version、outdated state 回传给页面

它仍然不负责右侧详情区域。

右侧详情、保存、删除、批量确认、刷新分析等页面级行为仍由 `RandomQuestionBoardClient` 管理。

这样可以保证：

- 日历视图能力可以复用
- 滑动日期范围能力可以复用
- random question 的业务流程集中在一个专用容器里
- 页面主体不需要直接处理范围弹窗内部状态

这里的层级关系应明确理解为：

1. `CalendarStatusView` 是基础日历组件
2. `RandomDateRangeDialog` 是基础范围输入组件
3. 如果业务只需要“看日历”或“选范围”，可以单独使用这两个基础组件
4. 如果业务需要“带状态数据的日历视图 + 范围选择 + 请求接口 + 结果回填 + 页面联动”，就应该由业务方自己再封装一个 panel/container

换句话说：

- 通用层只沉淀“日历怎么显示”“范围怎么选”
- 业务层自己封装“状态怎么映射”“按钮点了做什么”“范围 apply 后怎么请求和回填”

## 8.5 RandomQuestionBoardClient usage

`RandomQuestionBoardClient` 使用日历区域时，只组合专用业务容器：

```tsx
<RandomQuestionCalendarPanel
  selectedDate={selectedDate}
  savedDates={generatedDateSet}
  plannedDays={plannedDays}
  activeSnapshotVersion={activeSnapshotVersion}
  onSelectedDateChange={setSelectedDate}
  onPlannedDaysChange={setPlannedDays}
  onPreviewStateChange={setPreviewState}
  onSnapshotVersionChange={setActiveSnapshotVersion}
  onPlannedDataOutdatedChange={setPlannedDataOutdated}
  onOpenPlanActions={() => setPlanActionsOpen(true)}
  onShowDetailsPanel={() => setActiveTopPanel('details')}
/>
```

页面主体保留的职责是：

- 当前选中日期
- 当前 planned days
- 当前 preview/detail/analysis state
- 保存、删除、替换、批量保存
- 刷新分析
- 右侧详情和信息面板
- 各类确认弹窗

页面主体不再直接关心：

- range dialog 是否打开
- range draft value 是什么
- range apply 后如何请求 plan-range
- 日历状态 Map 如何从 saved/planned 数据生成

## 8.6 Usage patterns for other business areas

其他业务使用时应按需求选择组合方式。

### Only use the range dialog

适用于筛选、报表、导出等场景。

业务方只接收一个日期范围，然后触发查询或导出。

这时它等价于一个“日期范围输入器”。

### Only use the calendar view

适用于发布日历、排期日历、任务日历等场景。

业务方提供：

- 当前选中日期
- 日期状态 Map
- 顶部主操作按钮

这时它等价于一个“带状态标记的月历视图”。

### Use both together

适用于“范围准备数据 + 日历逐日处理 + 批量处理”的场景。

这种模式下：

1. 日历仍然是主交互面板
2. 点击单个日期始终可以触发该日期的业务操作
3. 顶部主操作按钮用于打开范围选择或批处理
4. 范围选择只是准备一批日期相关业务数据
5. 范围 apply 后，业务方把结果映射回 `dayStates`
6. 用户继续通过日历逐日审查或操作
7. 批量操作是附加能力，不替代单日操作

这个模型是当前 random question 页面采用的模式。

但需要注意，这种模式本身不是基础组件直接提供的能力，而是业务方额外封装出来的能力。

也就是说，当你需要这样的完整流程时，应该自己再包一层类似 `RandomQuestionCalendarPanel` 的业务容器，而不是把这些逻辑继续塞回 `CalendarStatusView` 或 `RandomDateRangeDialog`。

---

## 9. Right-side panel design

右侧区域负责“当前选中日期”的具体信息和操作。

它不是全局配置区，而是一个跟随当前日期焦点变化的内容区。

## 9.1 Details

Details 面板负责当前日期的实际处理。

它展示：

- 当前日期
- 当前状态
- 当前日期的题组详情

状态包括：

- 已保存
- 已规划
- 单日预览
- 无数据

如果当前日期对应的是临时规划数据，用户可以直接在这里完成审查与保存。

## 9.2 Analysis

Analysis 面板负责展示全局分析结果。

它回答的是：

- 已经生成了多少天
- 还剩多少题目资源
- 还能继续支持多少天的新规划
- 当前题库的可用分布情况如何

这里同时承载一个重要控制动作：刷新分析。

刷新不是装饰性按钮，而是系统重新接受“最新全局结果”的明确入口。

## 9.3 Info

Info 面板负责承载补充性的资源信息，例如类别分布等。

它的作用不是直接驱动操作，而是帮助用户理解当前题库结构，辅助判断规划是否合理。

## 10. Backend product semantics

虽然本文档不讨论代码实现，但 random question 页面有一个非常关键的产品前提：

分析和规划不是轻量查询，而是基于一轮全量计算结果工作。

这意味着，系统必须给用户提供一份稳定的“当前可规划能力视图”，而不是每点一次按钮就重新从头算一遍。

这也是后端设计和前端体验必须绑定的核心原因。

## 10.1 Analysis as the base snapshot

当前系统把分析结果视为一份“当前可规划能力快照”。

这份快照包含两类信息：

1. 全局摘要
2. 当前仍可生成的所有候选题组

它不是只给出统计数字，而是作为后续规划、预览、保存的统一基础。

全局摘要中的统计指标也来自这份快照，而不是每次打开页面都重新扫描全量数据。

这包括：

- 已保存日期数量
- 已使用题目数量
- 剩余可用题目数量
- 可作为首题的剩余数量
- 预计还能生成多少天题组
- 各分类的剩余库存分布

其中，“预计还能生成多少天题组”并不是一个独立估算公式，而是当前快照中仍然保留的可用候选题组数量。

换言之，它表达的是：

- 基于当前这份快照
- 在不重算全库的前提下
- 还可以继续拿出多少组完整题组

## 10.2 Stable planning pool

一旦分析结果已经生成，后续的范围规划和单日预览都应从这份稳定结果中取数据。

这意味着：

- 范围规划不是再次全量计算
- 单日预览不是再次全量计算
- 它们只是对当前稳定候选池的不同取用方式

产品上，这能保证用户看到的规划结果在一段操作周期内可预测、可审查。

## 10.3 Save consumes availability

当用户保存一条或多条规划结果时，系统对待它的方式不是“重新生成一套全新世界”。

更准确的语义是：

- 当前候选池中有一部分题组已经被正式占用
- 这些题组不再可用于后续规划

因此保存行为会消耗当前可用候选池，而不是立刻整体重算。

这就是为什么用户可以：

- 先规划多天
- 然后只保存其中一天
- 剩余未保存的规划结果仍然继续有效

保存成功后，当前快照需要向前推进，但推进方式是轻量的“消费”，不是全量刷新。

它的业务含义是：

- 已保存日期列表要包含刚保存的日期
- 被保存的题组要从候选池中移除
- 剩余库存和预计可生成天数要跟随扣减
- 尚未保存的临时规划仍然保留，并继续基于同一份快照审查

这条规则非常关键。

如果保存一天就重建整份快照，那么用户手上正在审查的其他临时规划可能会全部变化，违反“规划期间保持稳定”的原则。

## 10.4 Refresh means accepting a new world state

只有在刷新发生时，系统才重新构建全局分析结果。

对用户来说，这代表：

- 之前那份稳定规划基础被替换
- 之后的预览和规划都要基于新的可用题组能力重新理解

所以刷新不是普通重载，而是“接受新的全局状态”。

刷新后的结果可以与刷新前不同，这是符合预期的。

因为它代表系统重新读取了正式保存数据和题库数据，并重新计算可用题组池。

因此刷新会结束旧快照语义：

- 旧的候选池不再作为后续规划基础
- 旧的统计摘要不再继续沿用
- 前端持有的临时规划如果来自旧快照，应被清空或视为过期

## 10.5 Clear means deleting saved reality

Clear 的语义与普通取消预览不同。

取消预览只是放弃前端临时结果；Clear 是删除某一天已经正式保存的题组数据。

因此 Clear 影响的是正式业务事实：

- 这一天不再拥有已保存题组
- 这些题目不再被这一天占用
- 后续可规划能力可能发生变化

这会破坏当前快照对题组池的假设。

原因是，当前快照中的候选池和统计摘要，是基于“哪些题目已经被正式使用”构建出来的。删除某天已保存题组后，正式使用集合发生变化，原快照就不能再准确表达新的可规划能力。

所以 Clear 成功后必须重新构建分析快照。

这里与 Save 的处理不同：

- Save 是从当前候选池中消费一组已知题组，可以安全地轻量推进快照
- Clear 是把正式使用集合回退了一部分，会改变题组池的基础输入，必须重建快照

这条规则保证 Clear 后页面不会继续展示已经删除的日期，也不会继续沿用错误的剩余题量和预计可生成天数。

## 10.6 Pending plan guard before destructive changes

当页面存在尚未处理的临时规划数据时，用户不能直接执行会重建快照的破坏性操作。

当前最典型的场景是：

- 页面上已有一批临时规划
- 用户尝试 Clear 某一天已保存题组

系统应先阻止 Clear，并提示用户回到当前临时规划进行处理。

这个拦截不是为了保护删除接口本身，而是为了保护用户正在审查的计划。

因为 Clear 会导致快照重建，而快照重建代表旧规划基础失效。如果允许用户在未处理临时规划时直接 Clear，就会造成：

- 用户眼前的临时规划仍然显示
- 但它们依赖的可规划能力已经被替换
- 后续保存时容易出现过期、冲突或语义不一致

因此，存在 pending plan 时，页面应明确要求用户先处理当前计划，再允许执行 Clear。

这与“页面同一时刻只维护一批临时规划”的原则一致。

## 11. Date assignment rule for range planning

范围规划的结果并不是简单把候选题组机械填进用户选中的每一天。

当前业务规则是：

- 用户选择一个起止日期区间
- 系统尽可能为该区间安排规划结果
- 如果区间中的某些日期已经被正式占用
- 系统会继续向后顺排，直到尽可能满足目标天数

因此用户选择的是“希望从哪里开始，规划多长窗口”，而不是绝对锁死每一天必须落在原区间内部。

这是一条非常重要的业务语义：

- 目标是尽可能多生成连续规划结果
- 而不是制造大量冲突提醒然后让用户手工处理

## 12. Version consistency and stale-state handling

## 12.1 Why version consistency matters

由于分析结果本身是一份稳定快照，所以页面必须知道：

- 当前用户手上的规划结果，是不是仍然基于同一份分析结果

如果不是同一份，就不能继续假设这批计划安全可用。

## 12.2 User-facing rule

当系统发现当前页面持有的规划数据已经落后于最新分析结果时：

- 当前规划数据被视为过期
- 与规划相关的保存和生成动作会被限制
- 页面明确提示用户刷新

重点在于：

- 系统不自动悄悄替换用户眼前的数据
- 系统也不让用户在过期结果上继续保存
- 而是明确提示：请刷新，再继续操作

## 12.3 Error presentation principle

版本不一致属于“需要用户处理的状态”，不是一次性的瞬时报错。

因此它的提示应遵循：

- 提示必须持续存在，直到用户刷新成功
- 提示区应与普通错误区保持统一位置
- 提示区中附带明确辅助操作，例如刷新

这样用户才不会遇到：

- 按钮仍然被禁用
- 但页面上已经看不到原因

## 13. Concurrency and safety

保存行为需要在并发情况下仍然可控。

对产品来说，这意味着：

- 同一批规划结果不能被重复消费
- 多个保存操作不能同时把同一组数据都当成可用
- 批量保存不能在中途与其他写操作互相覆盖

因此系统必须保证：

- 保存是受保护的
- 候选池的消费是串行可判定的
- 用户看到的结果与最终落库结果保持一致

用户未必感知到锁和缓存，但会直接感知到结果是否稳定、是否出现重复、是否出现“刚才还能存现在不能存”的混乱体验。

除了保存，删除已保存题组也属于会改变全局可规划能力的写操作。

因此它同样需要被视为受保护的状态变更：

- 删除不能与保存并发破坏同一份快照语义
- 删除成功后必须让后续分析读取到新的快照
- 删除前必须避免用户仍持有未处理的临时规划

产品上最终要保证的是：用户看到的 Saved、Planned、Analysis 三类状态始终来自同一套可解释的业务事实。

## 14. Mobile design requirement

本页所有关键交互都必须移动端可用，尤其包括：

- 月历查看
- 日期切换
- 范围规划弹窗
- 滑块拖动
- 批量操作弹窗
- 错误提示与刷新恢复

移动端不是桌面端缩小版，而是完整支持同一任务闭环的界面。

因此设计时始终遵循：

- 弹窗必须拥有足够层级并独立遮罩页面
- 触控操作应优先保证可达性与连续性
- 状态提示不能依赖 hover 或隐性反馈
- 横向空间不足时，信息密度要通过布局压缩解决，而不是丢失语义

## 15. Final design conclusion

当前 random question 页面的设计已经形成以下稳定结论：

1. 月历负责日期焦点与状态概览
2. 范围规划通过独立弹窗完成
3. 范围只是输入，规划结果才是页面主状态
4. 页面同一时刻只维护一批临时规划结果
5. 用户可以逐天审查、逐天保存，也可以批量保存
6. 全局分析结果作为稳定规划基础存在
7. 保存消耗当前可用规划能力，但不立即重建整个世界
8. 删除已保存题组会破坏当前题组池假设，因此成功后必须重建分析快照
9. 存在未处理临时规划时，应先处理计划，再允许执行会重建快照的破坏性操作
10. 刷新表示接受新的全局分析状态
11. 版本不一致时必须显式提示并要求用户刷新
12. 所有关键交互都必须同时兼顾桌面端与移动端

这些结论不是局部 UI 选择，而是围绕真实业务需求逐步沉淀出来的统一设计。

后续功能继续迭代时，应优先保持这些设计原则稳定，再讨论局部交互或视觉细节的变化。
