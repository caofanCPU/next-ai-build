# Next.js / Vercel 打包优化排查总结

本文记录 `apps/ddaas` 首页、内容页、定价页和 API route 的函数体积排查过程，以及后续开发时需要遵守的依赖边界。

## 1. 背景

本次排查起点是首页 Vercel function 体积超过 30MB，怀疑来源包括 Clerk 用户组件、fingerprint provider、匿名用户 context 和 `backend-core`。

实际结论：

- 页面函数体积异常的主要原因是 server layout/page 的静态 import 链把 `backend-core/database`、`auth/server`、`context` 引入页面 route。
- 一旦 route import 到 Prisma，Next output file tracing 会把 Prisma runtime 和 query engine trace 到该函数。
- `@windrun-huaiin/backend-core/lib` 作为重 barrel 入口会把轻配置和重后端 helper 混在一起，是一次明确的误导入来源。

## 2. 排查方法

Next build 后，每个 App Router route/function 会生成 `.nft.json`：

```txt
apps/ddaas/.next/server/app/**/page.js.nft.json
apps/ddaas/.next/server/app/**/route.js.nft.json
```

这些文件记录该 function 部署时需要携带的文件。统计方式：

```bash
node - <<'NODE'
const fs = require('fs');
const path = require('path');
const root = 'apps/ddaas/.next/server/app';
const files = [];
function walk(dir) {
  for (const ent of fs.readdirSync(dir, { withFileTypes: true })) {
    const p = path.join(dir, ent.name);
    if (ent.isDirectory()) walk(p);
    else if (p.endsWith('.nft.json')) files.push(p);
  }
}
walk(root);
for (const p of files) {
  const trace = JSON.parse(fs.readFileSync(p, 'utf8'));
  const base = path.dirname(p);
  let bytes = 0;
  for (const f of trace.files) {
    try {
      bytes += fs.statSync(path.join(base, f)).size;
    } catch {}
  }
  const prisma = trace.files.filter((f) => f.includes('@prisma')).length;
  console.log(`${(bytes / 1024 / 1024).toFixed(2)}MB @prisma=${prisma} ${p}`);
}
NODE
```

判断重点：

- `@prisma=0`：该 route 基本没有携带 Prisma runtime。
- `@prisma>0`：该 route import 链触达 Prisma/database。
- 不能只看源码是否直接 import Prisma；barrel export、server helper、共享 context 都可能间接引入。

## 3. 本次发现的问题

### 3.1 `backend-core/lib` 重 barrel 污染页面

原 `packages/backend-core/src/lib/index.ts` 同时导出：

```ts
export * from './money-price-config';
export * from './money-price-helper';
export * from './fingerprint-config';
export * from './credit-init';
export * from './stripe-config';
export * from './upstash-config';
export * from './upstash';
```

其中 `fingerprint-config` 是轻配置，但 `money-price-helper`、`stripe-config` 会触达 auth/db/Stripe。页面只想取 `fingerprintConfig`，却从 `@windrun-huaiin/backend-core/lib` 导入，导致 blog 页面被 trace 到 Prisma。

处理结果：

- 删除 `@windrun-huaiin/backend-core/lib` package export。
- 删除 `packages/backend-core/src/lib/index.ts`。
- 新增明确 subpath exports：

```txt
@windrun-huaiin/backend-core/config/fingerprint
@windrun-huaiin/backend-core/config/money-price
@windrun-huaiin/backend-core/pricing/server
@windrun-huaiin/backend-core/credit/server
@windrun-huaiin/backend-core/stripe/server
@windrun-huaiin/backend-core/upstash/server
```

### 3.2 `CreditPopover` 在首页 layout 中直接查 DB

原链路：

```txt
home/layout.tsx
  -> layout.heavy.tsx
    -> CreditPopover
      -> getOptionalServerAuthUser
      -> creditService / subscriptionService
      -> buildInitUserContextFromEntities
      -> Prisma
```

这会让首页 page function 携带 Prisma。

处理结果：

- 新增 `apps/ddaas/src/app/api/user/credit-overview/route.ts`，把 DB 查询迁入 API route。
- 新增 `apps/ddaas/src/components/credit-popover-client.tsx`，首页只渲染 client shell 并 fetch API。
- `layout.heavy.tsx` 改为使用 `CreditPopoverClient`。

### 3.3 `pricing` 页面直接读取用户上下文

原 `pricing/page.tsx` 直接调用：

```ts
getMoneyPriceInitUserContext()
```

该 helper 会读取 auth/context/db，导致 `/pricing` 页面函数携带 Prisma。

处理结果：

- 新增 `apps/ddaas/src/app/api/user/pricing-context/route.ts`。
- 新增 `apps/ddaas/src/app/[locale]/(home)/pricing/pricing-client.tsx`。
- `pricing/page.tsx` 只在 server 端构造纯展示数据 `buildMoneyPriceData()`，用户上下文由客户端 fetch。

## 5. 为什么 DB API 动辄 80MB

当前 DB API 的大包主要来自 Prisma，而不是业务代码。

典型拆分：

```txt
Prisma runtime             约 57.12MB
Prisma generated client    约 20.82MB
Next runtime               约 0.92MB
其它业务/依赖              通常 < 1.5MB
```

最大文件包括：

```txt
.prisma/client/libquery_engine-darwin-arm64.dylib.node
@prisma/client/runtime/query_engine_bg.*.wasm-base64.js
@prisma/client/runtime/query_engine_bg.*.wasm-base64.mjs
@prisma/client/runtime/query_compiler_bg.*
```

这说明：

- DB API 大包的绝对主因是 Prisma runtime/query engine trace。
- Stripe、Clerk、业务 service 不是主要体积来源。
- 多个 DB API route 会各自形成独立 serverless function trace；本地物理文件未必复制多份，但部署函数包会各自携带所需文件。

## 6. 后续使用建议

### 6.1 页面/layout 禁止 import 重后端入口

页面、layout、普通 Server Component 中禁止直接 import：

```txt
@windrun-huaiin/backend-core/database
@windrun-huaiin/backend-core/context
@windrun-huaiin/backend-core/auth/server
@windrun-huaiin/backend-core/pricing/server
@windrun-huaiin/backend-core/stripe/server
@windrun-huaiin/backend-core/upstash/server
```

除非明确接受该页面 function 携带 Prisma 或其它重依赖。

页面/layout 允许 import 的内容应满足：

- 纯配置对象。
- 类型。
- 不读取 `headers()` / `cookies()`。
- 不访问数据库。
- 不实例化 Stripe、Upstash、Prisma 等 SDK。

推荐：

```ts
import { fingerprintConfig } from '@windrun-huaiin/backend-core/config/fingerprint';
import { moneyPriceConfig } from '@windrun-huaiin/backend-core/config/money-price';
```

### 6.2 DB 数据走 API 或 server action 隔离

如果页面需要用户积分、订阅、价格上下文等 DB 数据，优先模式：

```txt
page/layout
  -> client shell
    -> fetch('/api/...')
      -> API route import backend-core/database
```

这样页面函数保持轻量，DB 成本落在按需 API route。

适合这种模式的场景：

- 导航栏积分弹窗。
- 定价页用户订阅状态。
- 登录后才需要展示的数据。
- 不要求首屏 HTML 必须包含完整用户态的数据。

### 6.3 谨慎使用 barrel exports

禁止把轻配置和重 server helper 放在同一个 public barrel 中。

错误示例：

```ts
export * from './fingerprint-config';
export * from './money-price-helper';
export * from './stripe-config';
export * from './database-helper';
```

推荐按运行时边界拆分：

```txt
config/*        纯配置，页面可用
*/server        server-only，可能重
database        DB-only
stripe/server   Stripe-only
upstash/server  Upstash-only
```

### 6.4 DB function 数量的含义

`DB function 数量` 指会 import Prisma/database 的独立 Next/Vercel serverless function 数量，不是数据库连接数，也不是代码里的函数数量。

例如：

```txt
/api/user/credit-overview
/api/user/pricing-context
/api/user/anonymous/init
```

这是 3 个 DB functions，每个都会 trace Prisma。

合并相关 API 可以减少部署函数数量，但不能降低单个 DB function 的 Prisma 体积。降低单函数大小仍需处理 Prisma runtime/engine。

### 6.5 Prisma 体积优化方向

后续如果要继续降低 DB API 单函数体积，应优先研究：

- Prisma generator / engineType 配置。
- 是否可以避免 trace 无关 provider runtime。
- 生产环境只需要 linux engine，不需要本地 darwin engine。
- Prisma Accelerate / driver adapter / `engineType = "client"` 等 serverless 方案。
- `outputFileTracingExcludes` 是否能安全排除无用 engine。

注意：不要盲目 exclude Prisma 文件。缺少必要 engine 会导致线上查询运行时报错。

## 7. 每次新增功能的检查清单

- 新增页面或 layout 后，检查对应 `.nft.json` 是否出现 `@prisma`。
- 新增 API route 后，确认它是否真的需要 DB；如果不需要，避免 import `backend-core/database/context/auth/server`。
- 新增 public package export 时，确认是否混入重 server helper。
- 价格、积分、用户上下文这类数据优先走 API/client shell，而不是直接放进全局 layout。
- build 后对比关键 route 体积：

```txt
home/page
pricing/page
blog/page
legal/page
api/user/*
api/stripe/*
api/webhook/*
```
